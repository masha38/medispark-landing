# 메디스파크 랜딩페이지 프로젝트 인수인계서

> 이 문서는 Antigravity, Codex, Claude, Gemini 등 어떤 AI 또는 개발자도 공용으로 사용할 수 있는 프로젝트의 최우선 인수인계 문서다.
> 새 작업자는 반드시 이 문서를 가장 먼저 읽고, 아래의 문서 우선순위와 확정 정보를 기준으로 작업한다.

## 1. 프로젝트 개요

- 프로젝트명: 브레인시티 메디스파크 분양 홍보 랜딩페이지
- 목적: 분양 정보 전달, 전화 상담 유도, 방문예약 및 고객 상담 DB 수집
- 현재 형태: 별도 빌드 과정이 없는 단일 정적 HTML 랜딩페이지
- 메인 실행 파일: `index.html`
- 고객 상담 연락처: `010-4896-1691`
- 개발 담당자 및 DB 기록값: `장문희`
- 운영 대상 타입: `84A`, `84B`, `101`
- 삭제된 타입: `59A`, `59B` (분양 마감 정보이므로 다시 노출하지 말 것)
- 기준 날짜: 2026-06-27

## 2. 새 AI가 읽어야 하는 문서 순서

새 AI 또는 개발자는 아래 순서대로 파일을 읽는다.

1. `안티크래피틱_인수인계.md`
   - 프로젝트의 현재 상태, 확정 정보, 미완료 작업과 주의사항을 확인한다.
2. `index.html`
   - 실제 운영 화면과 동작의 최종 기준이다. 문서와 코드가 다르면 우선 현재 코드를 확인하고 사용자에게 차이를 알린다.
3. `메디스파크_랜딩페이지_최종기획서.md`
   - 페이지 구성, 마케팅 전략, 문구와 전환 설계의 기준 문서다.
4. `앱스크립트_명세서.md`
   - 상담 신청부터 구글시트 저장 및 전화 연결까지의 연동 규칙을 확인한다.
5. `apps_script.gs`
   - 구글 Apps Script에 배포할 실제 서버 측 코드다.
6. `구글시트_컬럼_샘플.csv`
   - 구글시트의 열 이름과 저장 순서를 확인한다.
7. `메디스파크_홍보자료.pdf` 및 `hero_render/`
   - 분양 정보의 원본 자료와 랜딩페이지 이미지 자산이다.

참고용 파일:

- `landing_form_example.html`: 상담 폼 연동 구조를 따로 확인하기 위한 예제다. 실제 운영 파일은 아니다.
- `메디스파크_랜딩페이지_기획서.md`: 이전 기획 문서다. 최종기획서와 충돌하면 `메디스파크_랜딩페이지_최종기획서.md`를 우선한다.

## 3. 현재 구현된 기능

- 밝고 세련된 도시형 프리미엄 디자인
- 데스크톱 및 모바일 반응형 레이아웃
- 상단 고정 헤더와 모바일 하단 고정 상담 바
- 메인 비주얼, 핵심 수치, 입지, 교통, 단지 배치, 평면 타입, 미래가치 섹션
- `84A`, `84B`, `101` 타입 탭과 평면 이미지
- 메인 상담 폼, 최종 문의 폼, 팝업 상담 폼
- 이름, 연락처, 관심 타입, 상담 유형, 개인정보 동의 입력
- 전화 버튼의 `tel:01048961691` 연결
- 상담 DB 전송 성공 후 `010-4896-1691`로 전화 연결
- 접속 경로, 기기 유형, 사용자 환경 정보 수집
- DB에 개발 담당자 `장문희`를 숨은 값으로 기록

## 4. 확정 데이터와 변경 금지 사항

사용자의 별도 지시가 없는 한 아래 값을 유지한다.

| 항목 | 확정값 |
|---|---|
| 고객 연락처 | `010-4896-1691` |
| 전화 연결값 | `01048961691` |
| 개발 담당자 | `장문희` |
| 페이지명 | `메디스파크 랜딩페이지` |
| 운영 타입 | `84A`, `84B`, `101` |
| 제외 타입 | `59A`, `59B` |
| 구글시트 탭 이름 | `leads` |
| 시간대 | `Asia/Seoul` |

주의:

- `59A`, `59B`는 분양 마감된 타입으로 페이지, 폼, 기획서에 다시 추가하지 않는다.
- 고객 화면에 `개발 담당자 장문희` 같은 내부 DB용 정보를 노출하지 않는다.
- 연락처 변경 시 화면의 표시 번호와 모든 `tel:` 링크, `PAGE_CONFIG.telNumber`, 관련 문서를 함께 변경한다.
- PDF의 분양 문구, 수치, 교통 계획 등은 게시 전에 시행·분양 주체의 최신 자료로 사실 확인한다.

## 5. 고객 DB 저장 구조

프런트엔드가 Apps Script로 보내고 구글시트에 저장하는 열 순서는 다음과 같다.

| 순서 | 필드 | 내용 |
|---:|---|---|
| 1 | `submittedAt` | Apps Script가 생성한 접수 시각 |
| 2 | `name` | 고객 이름 |
| 3 | `phone` | 고객 연락처 |
| 4 | `developerName` | `장문희` |
| 5 | `pageName` | `메디스파크 랜딩페이지` |
| 6 | `leadType` | 신청 버튼 또는 폼 위치 |
| 7 | `interestType` | `84A`, `84B`, `101`, 미정 |
| 8 | `consultType` | 전화상담 또는 방문예약 등 |
| 9 | `inflowSource` | UTM 또는 유입 경로 |
| 10 | `deviceType` | mobile, tablet, desktop |
| 11 | `userAgent` | 브라우저 환경 정보 |

구글시트 첫 행은 `구글시트_컬럼_샘플.csv`의 헤더와 동일하게 만든다.

## 6. 현재 미완료된 필수 설정

랜딩페이지 파일은 완성되어 있으나 실제 DB 저장은 아래 두 값을 설정하기 전까지 작동하지 않는다.

1. `apps_script.gs`의 `YOUR_SPREADSHEET_ID`
   - 실제 구글시트 주소의 문서 ID로 교체한다.
2. `index.html`의 `YOUR_GOOGLE_APPS_SCRIPT_WEB_APP_URL`
   - Apps Script를 웹 앱으로 배포한 뒤 발급된 실행 URL로 교체한다.

이 두 값은 계정별 비밀 또는 운영 설정이므로 현재 프로젝트에 임의로 넣지 않았다.

## 7. Apps Script 배포 순서

1. 구글시트를 새로 만들고 탭 이름을 `leads`로 지정한다.
2. `구글시트_컬럼_샘플.csv`의 첫 행을 시트 첫 행에 입력한다.
3. 구글시트의 문서 ID를 `apps_script.gs`의 `SPREADSHEET_ID`에 입력한다.
4. Apps Script 편집기에 `apps_script.gs` 전체 내용을 붙여 넣는다.
5. 웹 앱으로 새 배포하고 실행 사용자는 소유자, 접근 권한은 운영 목적에 맞게 설정한다.
6. 발급된 웹 앱 URL을 `index.html`의 `PAGE_CONFIG.scriptUrl`에 입력한다.
7. 실제 휴대전화에서 상담 신청, 시트 저장, 전화 연결을 차례로 테스트한다.

Apps Script를 수정한 뒤에는 기존 배포가 자동 갱신되지 않을 수 있으므로 반드시 새 버전으로 다시 배포하고 실행 URL을 확인한다.

## 8. 배포 전 필수 검수 목록

- 화면에 `010-4896-1691`이 표시되고 모든 전화 버튼이 같은 번호로 연결되는지 확인
- 폼에서 이름과 올바른 휴대전화 번호가 필수로 검증되는지 확인
- 개인정보 동의 없이 전송되지 않는지 확인
- 상담 신청 1회당 구글시트에 1행만 저장되는지 확인
- 저장된 열이 헤더 순서와 정확히 일치하는지 확인
- 저장 성공 후 모바일에서 전화 앱이 열리는지 확인
- `84A`, `84B`, `101`만 노출되고 `59A`, `59B` 또는 분양 마감 타입이 없는지 확인
- 모든 이미지가 배포 서버에서 정상 표시되는지 확인
- 모바일 360px 전후, 태블릿, 데스크톱 화면을 확인
- 실제 도메인에서 HTTPS가 적용되는지 확인

## 9. 상업 운영 전 보완 권장 사항

다음 항목은 아직 프로젝트에 완전히 적용되지 않았으며 운영 전에 검토해야 한다.

- 개인정보처리방침 전문과 보관 기간, 이용 목적, 파기 절차, 개인정보 처리 주체 정보
- 개인정보 동의 문구 옆 전문 보기 또는 별도 정책 페이지
- 광고성 정보 수신 동의가 필요한 마케팅을 할 경우 필수 동의와 분리된 선택 동의
- GA4, Meta Pixel, 카카오 등 광고 전환 추적 코드 및 이벤트 설계
- 상담 성공, 전화 클릭, 방문예약 클릭 이벤트 측정
- 스팸 및 반복 제출 방지, 서버 측 필수값 검증, 입력값 길이 제한
- 운영 도메인, 파비콘, SNS 공유 이미지, 검색 노출용 메타 정보
- 실제 사업 주체, 분양 주체, 주소, 면책 및 고지 문구
- 최신 분양 조건과 개발 계획에 대한 법무·현업 검수

## 10. 파일 및 자산 구조

```text
land/
├─ 안티크래피틱_인수인계.md             # 새 AI가 가장 먼저 읽을 문서
├─ index.html                         # 실제 랜딩페이지 최종 실행 파일
├─ apps_script.gs                     # 구글시트 저장용 Apps Script
├─ 앱스크립트_명세서.md                 # 연동 상세 명세
├─ 구글시트_컬럼_샘플.csv                # 시트 헤더와 샘플 데이터
├─ 메디스파크_랜딩페이지_최종기획서.md    # 최종 기획 기준
├─ 메디스파크_랜딩페이지_기획서.md        # 이전 기획 참고본
├─ landing_form_example.html          # 폼 연동 참고 예제
├─ 메디스파크_홍보자료.pdf               # 분양 원본 자료
└─ hero_render/                       # 페이지에서 사용하는 이미지 자산
```

`hero_render/` 폴더의 한글 파일명은 현재 `index.html`에서 직접 참조한다. 배포 환경이 한글 URL을 안정적으로 처리하는지 확인하고, 파일명을 바꾸면 HTML의 이미지 경로도 반드시 함께 바꾼다.

## 11. 새 AI에 전달할 시작 프롬프트

아래 문장을 새 AI의 첫 요청으로 사용할 수 있다.

```text
이 폴더는 브레인시티 메디스파크 분양 랜딩페이지 프로젝트입니다.
먼저 안티크래피틱_인수인계.md를 전체 읽고, 그 문서에 적힌 순서대로 index.html, 메디스파크_랜딩페이지_최종기획서.md, 앱스크립트_명세서.md, apps_script.gs를 확인하세요.
현재 고객 연락처는 010-4896-1691, 개발 담당자 DB 값은 장문희이며 운영 타입은 84A, 84B, 101입니다. 분양 마감된 59A, 59B는 다시 노출하면 안 됩니다.
기존 디자인과 사용자 변경사항을 보존하고, 작업 전 현재 코드와 인수인계서의 차이를 확인한 뒤 이어서 진행하세요.
```

## 12. 다음 작업자에게 권장하는 첫 작업

1. 실제 구글시트 ID와 Apps Script 웹 앱 URL을 사용자에게 받아 연동한다.
2. 상담 폼을 실제 휴대전화에서 제출해 DB 저장과 전화 연결을 검증한다.
3. 개인정보처리방침 및 사업자 고지 내용을 운영 주체에게 받아 페이지에 반영한다.
4. 광고 채널이 확정되면 전환 추적 도구와 이벤트를 설치한다.
5. 운영 도메인에 배포한 후 모바일 중심으로 최종 QA를 진행한다.
